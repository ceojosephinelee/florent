# biz-rules.md — Florent 비즈니스 규칙 명세 (v2)

> AI는 이 문서의 규칙을 코드에 정확히 반영한다.
> 규칙이 모호하거나 누락된 경우 임의로 판단하지 않고 반드시 질문한다.

---

## 변경 이력

| 버전 | 변경 내용 |
|---|---|
| v1 | 초기 작성 |
| v2 | 타임슬롯 중복선택 명확화, 반경 2km 확정, 예산 TIER 정의 추가, started_at → created_at 대체 명시, EXPIRED 제안 구매자 노출 정책 추가, CONFIRMED 요청 판매자 노출 정책 추가, 판매자 슬롯 자유 선택 명시, 구매자 홈 API 추가 |

---

## 1. 용어 정의

| 용어 | 정의 |
|---|---|
| 요청 (Request) | 구매자가 작성하는 큐레이션 요청서. 생성 후 48시간 유효 |
| 제안 (Proposal) | 판매자가 요청에 대해 제출하는 제안서. 생성 후 24시간 유효 |
| 예약 (Reservation) | 구매자가 제안 선택 + 결제 완료 시 생성되는 확정 상태 |
| 픽업 (Pickup) | 구매자가 예약 확정된 가게로 직접 수령 |
| 배송 (Delivery) | 구매자가 지정한 주소로 수령 |

---

## 2. 예산 TIER 정의

| TIER | 레이블 | 설명 |
|---|---|---|
| TIER1 | 작게 | 한 손에 쏙 |
| TIER2 | 보통 | 기본 꽃다발 |
| TIER3 | 크게 | 풍성하게 |
| TIER4 | 프리미엄 | 매우 풍성 / 고급 |

> UI 고정 문구 (예산 근처 반드시 표시): **"꽃집마다 가격 정책은 다를 수 있어요."**
> 가격 범위를 TIER로 한정하지 않는다. 판매자가 자유롭게 가격을 제시한다.

---

## 3. 타임슬롯 규칙

### 3.1 구매자 요청 — 타임슬롯

| 수령 방식 | 선택 단위 | 중복 선택 |
|---|---|---|
| 픽업 | 30분 슬롯 (10:00 ~ 20:00) | **가능 (복수 선택)** |
| 배송 | 오전(9~12) / 오후(12~18) / 저녁(18~21) | **가능 (복수 선택)** |

### 3.2 판매자 제안 — 타임슬롯

| 수령 방식 | 선택 단위 | 중복 선택 | 선택 범위 |
|---|---|---|---|
| 픽업 | 30분 슬롯 | **불가 (단 1개)** | 구매자 요청 슬롯과 무관하게 자유 선택 |
| 배송 | 오전 / 오후 / 저녁 | **불가 (단 1개)** | 구매자 요청 슬롯과 무관하게 자유 선택 |

> 예약은 시간 재고 개념이 아니라 **약속 확정**으로 단순화.
> 판매자의 슬롯 제시는 참고 정보이며, 재고 차감/중복 방지 로직 없음.

> UI 안내 문구 (판매자 제안 작성 화면): **"구매자가 요청한 시간을 고려하여 가능한 시간을 선택해주세요."**

---

## 4. 요청서 필드 규칙

- **목적 / 관계 / 분위기**: 추천 태그 칩 (복수 선택) + 직접 입력 (엔터로 태그화, 복수 가능)
- **예산**: TIER1 ~ TIER4 중 1개 선택
- **수령 방식**: 픽업 / 배송 중 1개
- **날짜**: 수령 날짜 1개
- **타임슬롯**: 3.1 규칙 적용 (복수 선택 가능)
- **장소**:
  - 픽업: 픽업 장소 (구매자 직접 입력, 카카오 주소 검색)
  - 배송: 배송 주소 (구매자 직접 입력, 카카오 주소 검색)

---

## 5. 요청 전송 규칙 (반경)

- 기준 좌표:
  - 픽업: 픽업 장소 좌표
  - 배송: 배송 주소 좌표
- 기준 좌표로부터 **반경 2km 이내** FLOWER_SHOP에만 요청 전송
- 거리 계산: Bounding Box 1차 필터 + Haversine 2차 필터 (서버 계산)
- **"반경 n km 내 전송됨" UI 표시 없음**

---

## 6. 요청 만료 (48시간)

- `request.expiresAt = request.createdAt + 48h`
- 만료 처리: 스케줄러 1~5분 간격으로 `OPEN → EXPIRED`
- 만료 후 판매자: 목록에서 **"만료" 배지** 표시, 제안 작성/제출 불가
- 만료 후 구매자: 상세 진입 가능 (히스토리), 신규 제안 수신/선택 불가

---

## 7. 제안 규칙

### 7.1 제안 작성 시작

- 판매자가 "제안서 작성하기" 버튼 클릭 → `PROPOSAL` 생성 (`status = DRAFT`)
- `PROPOSAL.created_at` = 작성 시작 시점 (`started_at` 별도 컬럼 없음, MVP)
- 작성 시작 즉시 구매자 요청 상세의 `draftProposalCount` 증가
- **UI 경고 문구 (필수 표시)**: "제안서 작성을 시작한 뒤 정해진 시간 내 제출하지 못하면 패널티(신뢰도 지수 하락)가 있을 수 있어요. 신중하게 버튼을 클릭해주세요." ← MVP에서 패널티 기능 미구현, 문구만 표시

### 7.2 제안 필수 필드

- `description` (상세 설명): **필수**
- `availableSlot` (가능 시간 단 1개): **필수**
- `price` (가격, 0 초과): **필수**

### 7.3 가격 정책

- 예산(TIER) 내로 제한하지 않음. 판매자가 자유롭게 제시
- **UI 안내 문구**: "요청서의 예산을 고려하여 가격을 정해주세요."

### 7.4 제안 제출 조건

- `status = DRAFT`인 제안만 제출 가능
- 요청이 `OPEN` 상태일 때만 제출 가능 (만료/확정된 요청에 제출 불가)
- 한 요청에 같은 가게가 중복 제안 불가

---

## 8. 제안 만료 (24시간)

- `proposal.expiresAt = proposal.createdAt + 24h`
- 만료 처리: 스케줄러가 `SUBMITTED / DRAFT → EXPIRED` (단, `SELECTED` 상태는 만료 처리 제외)
- 구매자 화면에서 **제안 카드/상세에 만료까지 남은 시간 표시** (필수)
- **EXPIRED 제안은 구매자에게 노출 (만료 배지 표시)**, 선택 불가

---

## 9. 구매자 제안 선택 UX

- 구매자 요청 상세에서:
  - **"n명의 판매자가 제안서 작성을 시작했어요"** 표시 (DRAFT 포함 카운트)
- 제안 목록에서:
  - `SUBMITTED` 상태 제안: 정상 노출
  - `EXPIRED` 상태 제안: **만료 배지와 함께 노출**, 선택 버튼 비활성화
  - **가격 비노출** (목록에서)
  - 제안 제목에 큐레이션 내용(concept_title) 표시
- 제안 상세에서:
  - 가격은 **맨 마지막**에 표시

---

## 10. Mock 결제

- 결제 수단 입력 UI 없음
- "결제하기" 버튼 → 확인 모달 → 즉시 성공 처리
- 성공 시 예약 확정으로 이동
- `PAYMENT.provider = MOCK`, `status = SUCCEEDED`
- `idempotency_key`: 클라이언트가 UUID로 생성하여 전송 (중복 결제 방지)

---

## 11. 예약 확정 및 미선택 처리

단일 트랜잭션 내에서 아래 순서로 처리:

1. `request.status = CONFIRMED`
2. 선택된 `proposal.status = SELECTED`
3. 나머지 SUBMITTED `proposal.status = NOT_SELECTED`
4. `RESERVATION` 생성 (`status = CONFIRMED`)
5. `PAYMENT` 생성 (`provider = MOCK`, `status = SUCCEEDED`)
6. 선택된 판매자에게 `RESERVATION_CONFIRMED` NOTIFICATION + OUTBOX_EVENT 저장

- **미선택 판매자 알림 없음** (앱 내 `NOT_SELECTED` 배지로만 표시)
- CONFIRMED된 요청에는 더 이상 제안 불가

---

## 12. 알림 발송 규칙

| 이벤트 | 수신자 | 알림 타입 | 발송 조건 |
|---|---|---|---|
| 요청 생성 | 반경 2km 내 판매자 (가게별) | `REQUEST_ARRIVED` | 요청 생성 트랜잭션과 함께 |
| 제안 제출 | 구매자 | `PROPOSAL_ARRIVED` | 제안 제출 트랜잭션과 함께 |
| 예약 확정 | 선택된 판매자 | `RESERVATION_CONFIRMED` | 예약 확정 트랜잭션과 함께 |

> 미선택 판매자에게는 알림을 보내지 않는다.
> 모든 알림은 DB 트랜잭션과 함께 NOTIFICATION + OUTBOX_EVENT로 저장 후 Worker가 FCM 전송.

---

## 13. 판매자 요청함 노출 규칙

| 요청 상태 | 판매자 앱 노출 방식 |
|---|---|
| `OPEN` | 정상 노출, 제안 작성 가능 |
| `EXPIRED` | **"만료" 배지** 표시, 제안 작성 불가 |
| `CONFIRMED` | **"마감" 배지** 표시, 제안 작성 불가 |

---

## 14. 위치 / 반경 규칙

- 반경: **2km 고정**
- 기준: 픽업 장소 또는 배송 주소 좌표
- 계산: Bounding Box 1차 SQL 필터 + Haversine 2차 서버 필터
- MVP: shop 수가 적다고 가정, 전체 조회 후 서버 필터링 허용
- PostGIS 확장: MVP 범위 외

---

## 15. 만료 처리 스케줄러

- 주기: 1 ~ 5분 간격 (`@Scheduled`)
- 처리 대상:
  - `OPEN` 요청 중 `expiresAt < now` → `EXPIRED`
  - `SUBMITTED` / `DRAFT` 제안 중 `expiresAt < now` → `EXPIRED` (단, `SELECTED` 제외)
- 클라이언트는 "남은 시간"을 표시하되, 최종 상태는 서버 값을 신뢰

---

## 16. S3 이미지 업로드 (제안서)

- 제안서 이미지만 MVP 포함 (꽃집 프로필 이미지 MVP 제외)
- 플로우:
  1. 클라이언트 → `POST /api/v1/images/presigned-url` 요청
  2. 클라이언트 → Presigned URL로 S3에 직접 PUT 업로드
  3. 업로드 완료 후 imageUrl을 제안 임시저장 API에 포함
- `PROPOSAL.image_urls_json`: S3 URL 목록, NULLABLE

---

- 실결제 PG 연동, 결제 수단 입력 UI
- 채팅
- 배송기사 / 물류 연동
- 실시간 타임슬롯 재고 관리
- 판매자 패널티 기능 구현 (문구 안내만)
- "반경 n km 내 전송됨" UI 표시
- "이미 지난 시간 회색 표시" 기능
- 꽃집 프로필 이미지
- Redis
- 자체 이메일/비밀번호 로그인
