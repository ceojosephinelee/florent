# api-decisions.md — API 설계 결정 이유

> API 설계에서 내린 결정과 그 이유를 기록한다.
> "왜 이렇게 설계했는가"를 남겨 AI가 일관성을 유지하도록 한다.

---

## 확정된 설계 결정

### [v2] GET /buyer/home 추가
- **결정**: 구매자 홈 전용 API 추가
- **이유**: 기존 API 조합으로는 프론트에서 여러 번 요청 필요. UX 개선을 위해 단일 API로 통합
- **내용**: OPEN 요청 최신 1건 + 읽지 않은 알림 최대 3건

### [v2] 제안 목록에 EXPIRED 노출
- **결정**: GET /buyer/requests/{id}/proposals 에서 SUBMITTED + EXPIRED 모두 노출
- **이유**: 구매자가 만료된 제안도 히스토리로 볼 수 있어야 함. 선택 버튼은 클라이언트에서 비활성화

### [v2] 판매자 요청 목록에 badgeLabel 필드 추가
- **결정**: status 외에 badgeLabel("진행중" | "만료" | "마감") 필드 추가
- **이유**: 클라이언트가 한국어 레이블을 직접 가공하지 않도록 서버에서 내려줌

### 구매자 타임슬롯 — 복수 선택
- **결정**: requestedTimeSlots 복수 선택 가능
- **이유**: 구매자 입장에서 가능한 시간대를 여러 개 제시해 판매자가 선택하는 방식이 UX상 자연스러움

### 판매자 타임슬롯 — 단수 + 자유 선택
- **결정**: availableSlot 단 1개, 구매자 요청 슬롯과 무관하게 자유 선택
- **이유**: 예약은 "시간 재고 관리"가 아닌 "약속 확정". 재고 차감/중복 방지 로직 없음.

### 가격 노출 위치
- **결정**: 제안 목록에서 가격 비노출, 상세에서만 노출 (응답 필드 맨 마지막)
- **이유**: 구매자가 가격보다 큐레이션 내용을 먼저 판단하도록 유도

### idempotencyKey
- **결정**: 클라이언트가 UUID 생성하여 전송
- **이유**: 중복 결제 방지. 나중에 실 PG 연동 시 동일 키 재사용 가능
